The technical problem

Every AI agent that crawls your site performs the same useless work: downloading hundreds of KB of HTML + assets, parsing the DOM, filtering CSS, deduplicating metadata (the title exists 6 times), handling 3 different encodings. To extract ~3,000 bytes of useful content.

This isn't an AI problem. It's a format problem. HTML was designed in 1993 for a human in a browser.

CBOR-Web v3.0 architecture

A single index.cbor file at the root. 7 top-level keys:

Practical integration

  • Minimal (30 min) — Serve an index.cbor with keys 0-2-6. Like creating a sitemap.xml.
  • Standard (1-2 days) — Pages (key 5) + navigation (key 4) + security (key 3). Full indexation in 1 GET.
  • Full (3-5 days) — Generative blocks, templates, T0/T1/T2 tiers with COSE signature.

Why CBOR and not JSON?

JSON is textual, non-deterministic, without byte strings or semantic tags. CBOR is an IETF standard (STD 94) from the same authors as JSON, used in WebAuthn, COSE, CWT, and IoT. Deterministic encoding guarantees identical bytes for the same content — reproducible hashes, reliable caches.

Ecosystem compatibility

CBOR-Web complements robots.txt, sitemap.xml, llms.txt, JSON-LD, OpenAPI. It adds the missing layer: complete, structured content in native binary. A single GET instead of a multi-page crawl.
